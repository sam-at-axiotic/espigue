# Shared memory and state management in multi-agent LLM systems — how agents read and write common context without contention, staleness, or context bloat.

## Introduction

Multi-agent systems built on large language models confront a convergence of hard constraints that existing architectures do not solve. Each LLM backbone operates within a static context window that resets after every interaction (Tan, 2026; Liu et al., 2025; Xu et al., 2025), preventing native accumulation of experience. When multiple agents collaborate on long-horizon tasks—enterprise workflows, iterative reasoning, continual learning—information volume compounds, and a single agent's context limitation becomes a team bottleneck. At the same time, coordinating action across agents incurs synchronization costs that grow with team size, and maintaining aligned role understanding across dozens of interactions is empirically difficult. Over long task horizons, even carefully aligned agents drift in their internal representations and interpretations, leading to miscalibration that no single memory write-back or coordinate reset can repair (Rath, 2026). These constraints are not independent; they interact in ways the literature is only beginning to characterize.

This review maps the methodological and empirical state of that landscape around two interlocking tensions: **(T1) whether memory architecture should be hand-engineered or learned**, and **(T2) whether coordination should be centralized or decentralized**. These tensions are not abstract—they determine whether systems scale, adapt to new tasks, and remain reliable under long horizons. The focal question guiding this review is: **How should LLM-based multi-agent systems manage memory, coordinate action, and scale collaboration without collapsing under context-window constraints, coordination overhead, or architectural drift—and what design choices (hand-crafted vs. learned, centralized vs. decentralized, structured vs. adaptive) best navigate these trade-offs?**

The evidence does not yet converge on a single answer. Instead, it reveals a problem-lattice: a set of interlocking tensions that cut across independent design axes. T1 pits approaches that impose structure through explicit schemas, hierarchies, and routing heuristics against approaches that learn compression and storage policies end-to-end via reinforcement learning or gradient descent. T2 asks whether multi-agent systems should route information through a designated hub or coordinator, or allow agents to negotiate state without fixed roles. These tensions interact—a highly structured memory is easier to manage centrally but harder to adapt; a learned memory policy is flexible but more opaque to coordinate across—and neither axis has a dominant solution yet. The field is increasingly mapping the trade-offs that define each cell of the lattice rather than declaring a winner.

The evidence falls into distinct strength bands. First, the foundational problem is established: context windows reset (Tan, 2026; Liu et al., 2025; Xu et al., 2025), memory is required to persist experience and enable long-horizon adaptation (Yang et al., 2026; Sun & Zeng, 2025; Xu et al., 2025), and information volume in multi-agent settings intensifies that constraint (Pan et al., 2026; Liu et al., 2025). Second, several design approaches have been tested and shown to yield measurable efficiency gains—context routing cuts token consumption by 25–47% without quality loss, and saturation dynamics in routing efficiency emerge around 2048 tokens per agent (Liu et al., 2025). Third, emerging theory provides formal bounds on token savings under specified conditions, and architectural diversity is documented across multiple teams (hierarchical layers, spatial DAGs, RL policies, and small-language-model components (Sun & Zeng, 2025; Ye, 2025; Yu et al., 2026; Zhang et al., 2026)). Fourth, coordination overhead—not context-window exhaustion alone—drives multi-agent performance plateaus (Kim et al., 2025; Li et al., 2026; Yang et al., 2025; Mao et al., 2026), yet the field recognizes unresolved mechanisms: whether architectural drift and Semantic Intent Divergence are primary failure modes (Acharya, 2026), and whether hand-crafted designs generalize to new task domains (Yuen et al., 2025; Yu et al., 2026).

This review is organized as a problem lattice. Rather than converging toward a single answer, the evidence reveals a field increasingly clear about what trade-offs define the problem space but not yet unified on how to resolve them. Each major section occupies a cell of this lattice, presenting the tensions, the evidence pulling in each direction, and the methodological gaps that prevent settlement. Three planted threads—concerning transfer across task domains, non-monotonic scaling with agent count, and saturation thresholds in context routing—will be developed and cashed out across sections, tying the lattice together into a coherent research agenda.

Scope note: This review covers persistent, agent-specific memory and coordination in transformer-based multi-agent systems. It excludes pre-transformer architectures, fine-tuning and adapter-based personalization, static retrieval-augmented generation divorced from agent adaptation, human cognitive memory, and prompt engineering techniques without persistent state. The term registry that follows defines core concepts used throughout; adherence to canonical terms allows precision where field jargon diverges. **Memory** is a persistent, updatable external or embedded store of agent-specific experience, observations, or structured knowledge enabling long-horizon adaptation and continual self-improvement. **Context routing** is the mechanism by which a system selects and marshals a subset of stored information into the context window according to task stage, agent role, or learned heuristics. **Coordination overhead** is the token, computational, or latency cost incurred when multiple agents synchronize state, negotiate roles, or resolve conflicts. **Architectural drift** is unintended divergence in internal representations, role interpretations, or decision-making processes of agents over long task horizons, leading to miscalibration and degradation. **Semantic Intent Divergence** is the divergence of goal interpretation and objective alignment among agents that leads to miscalibration and reduced coherence. These terms appear verbatim throughout; synonyms are abandoned to maintain precision.

## The persistence problem: why static context windows demand external memory

The foundational constraint is precise: each LLM invocation receives input within a fixed token window and produces output; after that invocation, no trace of the interaction persists in the model's internal state (Tan, 2026; Liu et al., 2025; Xu et al., 2025). For single-turn tasks—a question answered once, a document classified—this poses no obstacle. For agents tasked with iterative reasoning, long-horizon planning, or continual improvement, the reset is fatal. An agent cannot learn from its own past performance, recognize patterns in its failures, or build upon yesterday's partial solutions because there is nowhere for experience to accumulate within the model's weights or activations.

Memory solves this by externalizing experience into a persistent, updatable store (Yang et al., 2026; Sun & Zeng, 2025; Xu et al., 2025). Rather than asking the model to encode experience implicitly, memory systems ask it to write and retrieve explicit records: observations from past steps, decisions made, outcomes observed, and representations of its own performance. The mechanism transforms a stateless predictor into an agent with a biography. Recent work recasts the architectural problem as memory-centric adaptation: instead of asking how to fine-tune or prompt an LLM to adapt, the field increasingly asks *how to architect the memory store itself* such that it is efficiently written, selectively retrieved, and flexibly organized (Yang et al., 2026; Sun & Zeng, 2025; Xu et al., 2025). This reframing is consequential—it shifts focus from the backbone model to the system around it.

In multi-agent settings, the problem intensifies sharply because information volume does not scale linearly with agent count (Pan et al., 2026; Liu et al., 2025). A single-agent system must manage its own experience; information volume grows with task horizon but is bounded by the agent's internal output rate. In multi-agent systems, information volume is the sum of outputs from all agents. If ten agents each produce a status update, observations, and planned actions per interaction cycle, the system rapidly generates thousands of tokens of new experience. A context window of 4,096 tokens can access only a fraction of this history without aggressive compression or filtering. Information volume grows with agent count, and the density of relevant information per agent necessarily decreases—unless the system adopts some method to filter, route, or compress selectively. Systems like ScaleSim address this pressure via an abstraction—invocation distance—that guides memory management decisions for large agent simulations, surfacing a quantifiable structure (distance-based relevance) that memory systems can exploit (Pan et al., 2026; Liu et al., 2025).

This multiplication of volume reveals why context routing and memory architecture are not luxury optimizations but functional necessities. The problem is not merely making memory work; it is making memory work *at scale*, for multiple agents, under a token budget. The act of writing and retrieving from memory creates a secondary design problem: what information goes in, and how is it retrieved? A naive approach—dump every observation into memory, then retrieve everything for each query—quickly saturates the context window and produces retrieval noise. A more sophisticated approach imposes structure on memory: organize information by abstraction level, task stage, or semantic content; route queries to relevant subsets; compress or summarize older records. This is where the diversity of memory designs emerges, and where design choices begin to propagate consequences.

This establishes why memory is not optional and why its design is not a minor engineering detail. The remainder of this review traces how systems respond to this pressure, and where those responses diverge into the core tensions: which memory architecture survives contact with new domains, and which coordination approach scales without hitting diminishing returns driven by coordination overhead.

## Hand-engineered versus learned memory management: the generalization problem

The first major tension (T1) asks: Should memory architecture be hand-crafted for a task domain, or should it be learned end-to-end as an adaptive policy? The diversity of existing approaches reveals the breadth of the design space and exposes a shared vulnerability.

Structured memory architectures dominate the published approaches. H-MEM organizes information across four layers of increasing semantic abstraction, with layer-by-layer summarization and positional indices for retrieval (Sun & Zeng, 2025; Ye, 2025; Yu et al., 2026; Zhang et al., 2026). Agent-specific structured memory updates intrinsically from agents' own outputs, maintaining typed fields (goals, constraints, completed steps, next actions) directly from agent outputs. Spatial DAG memory (TME/TRIM) decomposes tasks into subtasks, represents them as directed acyclic graphs, and assembles compact subgraph prompts with intent vectors. All three exemplify hand-crafted design: humans decide what dimensions of semantic abstraction matter, which task decompositions reveal relevant structure, and how to index and retrieve information (Sun & Zeng, 2025; Ye, 2025; Yu et al., 2026; Zhang et al., 2026). The specificity is a strength—each approach is tailored to exploit known structure in the target domain. It is also a liability.

Hand-engineered designs do not transfer across task domains (Yuen et al., 2025; Yu et al., 2026). A memory schema built for multi-hop question answering (hierarchies of intermediate facts, question decomposition) does not fit protein-folding workflows (spatial coordinates, energy landscapes) or enterprise automation (role hierarchies, state machines). H-MEM's four-layer hierarchy assumes a specific task complexity profile; if the task is radically simpler or more complex, the layering scheme becomes suboptimal. Agent-specific schemas are built for a specific task; a new task with different constraints requires new schema design. Spatial DAGs assume tasks decompose into clear subtask hierarchies; open-ended exploration or negotiation tasks may not have this structure. Practitioners face a choice: either redesign the memory system for each new domain, or accept that the system's efficiency on a new domain will be poor because its structure is misaligned. This creates a transfer gap—one that grows as the field aims to deploy multi-agent systems across diverse tasks.

Learned memory management offers an alternative: replace hand-crafted heuristics with a reinforcement learning policy that automatically decides what to store, compress, and discard (Yuen et al., 2025; Yu et al., 2026). AgeMem exemplifies this approach, training an RL policy over a three-stage trajectory with injected distractors that penalizes redundant storage and uncontrolled context expansion. The learned policy subsumes human judgment about storage priorities and can, in principle, adapt to new domains by retraining or meta-training over a distribution of task types. However, the evidence supporting learned approaches remains sparse. AgeMem exists; its superiority over hand-engineered alternatives on out-of-domain tasks has not been demonstrated.

Between these poles sits partial automation: LightMem decomposes the system into small-language-model components with strictly typed prompt roles, allowing some hand-design (the component roles and types) while learning the routing between them. This hybrid position reduces the amount of domain-specific engineering required while avoiding the full burden of end-to-end learning.

Concrete efficiency gains from hand-engineered context routing are substantial. Role-aware context routing (RCR-Router) stores agents' intermediate outputs in shared structured memory and routes context under a per-agent token budget (Liu et al., 2025). Across three multi-hop QA benchmarks (HotPotQA, MuSiQue, 2WikiMultiHop), it achieves token savings of 25–47% without quality degradation: 4.91 quality at 3.77K tokens on HotPotQA and 4.83 quality at 1.24K tokens on 2WikiMultiHop (Liu et al., 2025). These gains are directly attributable to the hand-crafted routing logic—it routes only role- and stage-relevant context to each agent.

Yet these gains show saturation dynamics, setting up tension within T1 itself (Liu et al., 2025). As the per-agent token budget increases past approximately 2048 tokens, answer quality improves only sublinearly while token cost and runtime rise monotonically. Gains plateau beyond a critical threshold. This finding suggests that hand-crafted routing heuristics may hit an inherent efficiency ceiling. The saturation pattern indicates that routing decisions based on fixed rules (role matches task stage; agent type determines relevance) cannot adapt to the nonlinear relationship between available tokens and task difficulty. Beyond the threshold, additional tokens are wasted on redundant context because the routing rules lack the flexibility to discriminate fine-grained utility. If a heuristic routes only "relevant" context and quality plateaus despite higher budgets, the implication is that the heuristic's notion of relevance is incomplete or brittle—it cannot adaptively reprioritize based on feedback about what information actually matters for the task at hand. A learned policy, by contrast, could adjust its retrieval priorities dynamically and might overcome this saturation by discovering more efficient allocation strategies that respond to the actual value of additional context rather than applying static rules. But this is speculative; the evidence that learned policies avoid saturation is not present in the corpus. Whether saturation is a fundamental constraint intrinsic to the token-versus-quality trade-off or a limitation of current hand-engineered approaches remains unresolved.

The transfer gap and the saturation ceiling together define the unresolved core of T1. Hand-engineered systems excel at efficiency and interpretability within a single task domain but fail to generalize across domains. Learned approaches promise domain-agnostic adaptation and potential escape from saturation but have not yet demonstrated superiority in direct comparison, controlling for computational overhead and training data requirements. This section has planted both threads—transfer and saturation—which will be cashed in the future directions section.

## Efficiency bounds and formal guarantees in memory design

Where do hand-engineered heuristics and learned policies converge? A nascent theoretical question now has initial scaffolding: formal analysis of memory system efficiency. Early work on lazy artifact invalidation—the practice of deferring updates to derivative structures (indices, summaries, compressed versions) until they are actually accessed—has produced a theorem that bounds token savings under specified conditions (Parakhin, 2026). The bound is parameterized on coherence (how reliably derivative structures reflect base memory state) and refresh-cycle rates (how frequently base memory changes). Under these constraints, the system is guaranteed some minimum efficiency gain. The result is circumscribed: it applies to a specific mechanism, assumes coherence can be formally modeled, and requires predictable refresh schedules. Yet it demonstrates that the design space contains exploitable structure—it is not purely heuristic terrain.

This matters directly for resolving T1. Rather than choosing entirely between hand-engineered systems and full end-to-end learning, formal bounds suggest a hybrid path: a hand-engineered skeleton that respects provable efficiency constraints, combined with learned policies that optimize routing and artifact invalidation within those constraints. A learned policy could be trained to decide what to store and when to invalidate, but do so under regularization derived from the formal bound—penalizing storage decisions that violate the theorem's assumptions. Conversely, if a learned system observes efficiency gains that exceed the formal bound, that gap becomes a discovery: the bound's assumptions may be too conservative, or the system may have uncovered a structural property (coherence profile, refresh pattern) that exceeds typical cases. Hand-engineered systems gain interpretability and robustness by grounding design in formal principles; learned systems gain principled constraints rather than purely empirical objectives.

The evidence that this hybrid composition is superior to either approach alone remains absent. Only one source formalizes efficiency bounds; replication and extension are pending. But the theoretical pathway now exists, and it repositions the central question from "hand-crafted or learned?" to "how do learned policies and formal constraints compose?" This framing carries T1 forward into the next section, where future architectures are sketched against this dual foundation.

## Coordination overhead and the team-size ceiling

The second major tension (T2) asks: Should multi-agent systems use centralized coordination (a designated coordinator arbitrating state and routing information) or decentralized coordination (agents negotiating without fixed roles or central authority)? The answer is not settled, but the evidence increasingly reveals why the choice matters—and why it interacts with memory architecture.

Multi-agent performance does not scale monotonically with agent count (Kim et al., 2025; Li et al., 2026; Yang et al., 2025; Mao et al., 2026). Naive expectation suggests that adding an agent should add capability; the system should improve. Empirically, systems saturate or degrade as teams grow. The culprit is coordination overhead, not long-context failure alone. As agent count increases, the number of distinct state reconciliations, conflict resolutions, and message exchanges grows superlinearly. Each coordination step consumes tokens and latency. The trade-off is quantifiable: there is a synergy benefit (more agents = more diverse reasoning and coverage) and an overhead cost (more agents = more synchronization). The evidence is sharp: coordination overhead is the token, computational, or latency cost incurred when synchronization demands arise, and it dominates the benefits of added expertise. Overhead wins; teams hit a coordination ceiling (Kim et al., 2025; Li et al., 2026; Yang et al., 2025; Mao et al., 2026).

Centralized coordination designates one agent or coordinator service as the hub: other agents report outputs, constraints, and proposed actions to the center, which routes information and arbitrates conflicts. This design is easy to analyze—a single decision point, a single log of communications—and eliminates many sources of miscommunication. But it creates a single point of scaling failure: as agent count grows, the coordinator's context window and decision latency become bottlenecks.

Decentralized coordination offers one response, exemplified by AgentNet, which removes fixed roles and central coordinators, allowing agents to communicate and negotiate outcomes directly through message passing or consensus (Kim et al., 2025; Li et al., 2026; Yang et al., 2025; Mao et al., 2026). The claim is that eliminating the bottleneck reduces latency and token consumption per coordination cycle. Whether decentralized coordination genuinely avoids the overhead ceiling or merely shifts it remains contested. AgentNet is described but not benchmarked against centralized baselines in this corpus. The direction is clear; the verdict is not.

Collaborative RL (CoMAM) represents a hybrid approach: instead of hand-designating coordination roles or leaving coordination to local negotiation, train a joint RL policy that simultaneously optimizes each agent's behavior and the credit assignment connecting individual actions to team outcomes (Kim et al., 2025; Li et al., 2026; Yang et al., 2025; Mao et al., 2026). This treats coordination as a learning problem, not a design problem. Like learned memory management in the memory domain, it substitutes learning for human judgment. The scalability to large teams (15+ agents) is unexplored; the corpus does not report beyond seven agents, and even that reveals gaps.

Here emerges the interaction between T1 and T2. If memory architecture is hand-crafted (structured memory), agents must fit their reads and writes to a fixed schema. Coordination must then work within that schema's constraints. If memory is learned (learned memory management), agents can adapt their storage to suit their coordination needs. Conversely, if coordination is centralized, a single coordinator can arbitrate who accesses what memory and enforce consistency. If coordination is decentralized, agents must resolve memory conflicts through negotiation, or else accept eventual inconsistency. The tensions are not independent.

The deepest unresolved finding concerns the mechanism linking memory capability and team size. The interaction is established only up to seven-agent teams, and the causal mechanism remains opaque (Wu et al., 2026). Intuition suggests that richer, more accessible memory of team members' states and reasoning could reduce the communication overhead of coordination by making shared understanding implicit: if Agent A can query shared memory to learn what Agent B decided and why, A might not need to send a message to B. Yet the evidence does not isolate the direction. Do larger teams require richer memory architectures? Do richer memories enable larger teams? Does the benefit plateau beyond seven agents? Does memory capability's value depend on centralized versus decentralized coordination, or is it invariant to that choice? The corpus establishes the correlation but not the mechanism. This planted thread about scaling—that performance does not proceed monotonically with agent count and that the mechanisms governing scaling beyond seven agents remain unmapped—will be developed in future directions.

## Alignment and stability: semantic intent and architectural drift

As multi-agent systems extend to longer horizons and more agents, two failure modes emerge that are not purely engineering problems—they concern whether agents maintain coherent purpose and stable internal representations over time.

Semantic Intent Divergence is the first: agents develop incompatible or misaligned understandings of task objectives, preventing coordinated progress (Acharya, 2026). In an enterprise workflow where a human specifies a goal once and hands it to a multi-agent system, the initial alignment is correct. But as agents interact, each may interpret the goal slightly differently based on local observations, role, or interaction history. One agent understands "complete a purchase" as "execute the transaction," while another understands it as "verify fraud risk first." The goals are compatible in intent but incompatible in execution sequence. A Semantic Consensus Framework addressing this failure mode achieves 100% workflow completion versus 25.1% for the next-best baseline (Acharya, 2026). This result is single-source: one paper, one framework, one set of benchmarks. The effect size is dramatic, but the evidence is thin. The mechanism by which consensus on intent is maintained—whether through periodic alignment updates, role-specific objective refinement, or mutual belief validation—is not dissected. The study is limited to enterprise workflows, and the generality of the result to other multi-agent settings is unknown.

Architectural drift is the second: over long task horizons, agents' internal representations, role interpretations, and decision-making processes diverge unintentionally, leading to miscalibration and performance degradation (Rath, 2026). Unlike Semantic Intent Divergence, which is a goal-level misalignment, architectural drift is a creeping incoherence in how agents represent and act on their models of the task. An agent's internal understanding of what a successful state looks like may shift as it observes outcomes, and that shift may be untracked and incompatible with other agents' drifting models. Architectural drift correlates with performance degradation, but causal mechanisms remain partially speculative absent controlled ablation studies isolating them (Rath, 2026). The sources of drift are unknown: whether it stems from inconsistent observation signals causing differential agent updates, from disparate learning from mistakes, from lack of explicit alignment signals, or from memory itself—an agent that retrieves and replays old information may drift from its original role interpretation if that information subtly reframes the agent's purpose.

Both failures interact with memory and coordination in ways not yet integrated into the design landscape. If agents hold semantic intent divergently, they require more coordination overhead to resolve incompatibilities—more messages, more negotiation, more token consumption. If architectural drift is undetected, memory systems may store information based on misaligned representations, causing later retrieval to fail or to activate wrong inferences. Whether shared structured memory reduces Semantic Intent Divergence by making each agent's interpretation explicit and comparable, or whether it persists despite memory access as a consequence of fundamental task ambiguity, remains open. Whether architectural drift can be prevented by formal constraints on how memory is updated and compressed, enforcing consistency with initial role definitions, or whether it is an inevitable consequence of learning without explicit regularization, is equally unresolved.

The field is weakest here. The evidence is recent, exploratory, and mechanism-sparse. Claims about Semantic Intent Divergence and architectural drift must be read as hypotheses about failure modes rather than as established root causes, and solutions like the Semantic Consensus Framework must be read as evidence of concept rather than as established practice. Neither failure mode has been integrated into the broader design landscape: the field has not yet asked whether hand-engineered or learned memory management better resists Semantic Intent Divergence, whether centralized coordination prevents it or risks imposing misalignment, or how memory architecture and coordination design jointly affect drift rates over long horizons.

## Open questions and future directions

The problem-lattice framework—two core tensions (T1 and T2) and the empirical gaps within and between them—generates a concrete research agenda. This section cashes in the planted threads and maps the next steps required to move from trade-off recognition toward principled system design.

**Can automated methods or meta-learning close the transfer gap in memory management?** Hand-engineered structured memory does not transfer across domains (Yuen et al., 2025; Yu et al., 2026). Each new task requires redesign. The evidence for learned policies (AgeMem's RL approach) is recent and not yet tested on transfer. The crucial experiment is direct comparison: train an RL memory policy on one domain, freeze it, and evaluate it on held-out domains where it was not trained. Does it maintain efficiency, or does it collapse? If RL policies transfer better than hand-engineered designs, that evidence would strongly favor T1's "learned" arm. If they do not transfer, the field must explore why—is the RL objective itself domain-specific? Do agents need task-specific priors?

An alternative path is meta-learning: train a system to *generate* memory schemas automatically given a task description, rather than asking it to select actions within a fixed schema. This has not been demonstrated but is plausible; the mechanism would be a meta-learner that observes task structure and outputs a memory organization suited to it. Testing this requires defining what "suited to" means operationally (e.g., token efficiency, retrieval latency, answer quality on held-out examples). A third avenue is in-context adaptation: given a few examples of a new task, can a system rapidly infer and deploy appropriate context routing and compression policies without retraining?

**What mechanisms govern the scaling boundary beyond seven agents, and how can systems avoid or raise the coordination ceiling?** Scaling does not proceed monotonically with agent count; it saturates due to coordination overhead (Kim et al., 2025; Li et al., 2026; Yang et al., 2025; Mao et al., 2026). But the evidence stops at seven agents (Wu et al., 2026). The field does not know: Does the ceiling hold at 15 agents? 50? Does it depend on task structure—rising for some tasks and falling for others? What is the interaction between memory capability and team size?

Isolating this requires controlled experiments. Fix task complexity, vary team size from 2 to 20 or more, measure coordination overhead token consumption and marginal capability gain at each step. If overhead grows monotonically and capability gain falls sublinearly, coordination is driving saturation. If overhead plateaus or falls as teams grow (e.g., because agents learn efficient coordination codes), the ceiling might be raised. Second, test whether richer memory architectures (more storage capacity, faster retrieval, learned routing) allow teams to grow larger before saturation hits. This directly tests the T1–T2 interaction.

For decentralization specifically: AgentNet and role-free coordination are proposed as responses to the coordination ceiling, but they lack side-by-side benchmarking against centralized coordinators. Is decentralization intrinsically lower-overhead, or does it merely distribute overhead differently? The answer requires controlled ablation: same team, same task, one run with a central coordinator and one fully decentralized, measuring total token and latency costs. Do centralized and decentralized coordination have identical scaling profiles, or does memory capacity help one regime more than the other?

**How do learned memory policies and formal efficiency bounds combine?** Token savings from lazy artifact invalidation are formally bounded (Parakhin, 2026). Learned memory policies (AgeMem) make adaptive storage decisions (Yuen et al., 2025; Yu et al., 2026). The field has not asked: can an RL policy be trained to respect formal bounds while learning optimal invalidation timing? More broadly, can formal constraints guide the reward function of a learning policy, such that learned behavior is guaranteed to meet minimum efficiency thresholds while maximizing quality?

This requires formalizing the interaction: a learned policy that minimizes (quality loss + constraint violation) where constraints derive from formal analysis. Testing this requires defining task suites where formal bounds are computable and learned policies are trainable, then measuring whether the combined system outperforms either component alone. Additionally, investigate whether formal bounds can accelerate learned policy training; a policy that respects provable efficiency bounds may converge faster and generalize better than one trained without such structure. The payoff is a design methodology that combines the interpretability and provability of hand-engineered systems with the adaptability of learned policies.

**Is centralized or decentralized coordination fundamentally superior, or does it depend on task structure?** T2 is framed as a dichotomy, but the evidence suggests it is task-dependent. Some workflows (sequential, with clear phases) may naturally centralize. Others (exploratory search, with agents working in parallel on disjoint problems) may benefit from decentralization. The field has not systematized this. The next step is task taxonomy: characterize what properties (branching factor, communication frequency, role heterogeneity) predict whether a team should use centralized or decentralized coordination. Operationally: specify three or more diverse task suites, run each with both architectures, measure efficiency and quality, and ask whether any task property predicts which architecture wins. The hypothesis is that tasks with natural hierarchies may favor centralized coordination, while tasks with distributed dependencies favor decentralized coordination. This is not a hunt for a single winner but a characterization of the conditions under which each excels.

**What are the causal levers driving architectural drift and Semantic Intent Divergence, and how can they be prevented?** Both failure modes are identified but mechanistically opaque (Acharya, 2026; Rath, 2026). Architectural drift correlates with performance loss, and Semantic Intent Divergence is solved by a consensus framework, but the framework's components are not isolated and ablated.

Responsible next steps are narrow and mechanistic. For Semantic Intent Divergence: controlled experiments where goal statements are progressively obscured or ambiguous, measuring correlation with workflow failure. Identify whether misalignment is caused by inadequate goal specification, by different agents interpreting the same goal statement differently, or by agents updating their goals as they learn. Interventions can then be designed—explicit goal negotiation, goal clarification before planning, frozen goal statements—and their effects measured. The consensus framework can be ablated to isolate which components are necessary (periodic realignment, mutual belief validation, objective re-specification).

For architectural drift: instrument agents to log their internal role understanding and decision heuristics at regular intervals, and measure divergence over time. Does divergence increase with the number of observations? With memory capacity (more memory, more opportunity for drift)? With the heterogeneity of agent experience? Once drivers are identified, test interventions: periodic explicit alignment signals from a coordinator, shared memory of role definitions that all agents update together, or formal constraints on how agents can update their internal models. Test whether periodic memory coherence audits—explicit checks that agents' stored models of task state are compatible—prevent or mitigate drift without imposing prohibitive coordination overhead.

The evidence for these mechanisms is emerging and speculative. The gap is large: current evidence is correlational for drift and single-domain for semantic intent divergence. Multi-domain studies, mechanistic ablations, and causal inference experiments are required to move from observation to actionable design principles.

## Conclusion

This review has presented multi-agent LLM systems as a problem lattice organized around two core tensions: **(T1) whether memory should be hand-engineered or learned**, and **(T2) whether coordination should be centralized or decentralized**. These tensions are not abstract. They determine scalability, transferability, and reliability in practice.

The foundational constraint is solid: static context windows reset after each invocation (Tan, 2026; Liu et al., 2025; Xu et al., 2025), memory is necessary for persistence (Yang et al., 2026; Sun & Zeng, 2025; Xu et al., 2025), and multi-agent settings intensify information volume pressure. All design choices downstream must contend with this. The field has developed instruments to measure trade-offs—efficiency metrics for context routing, saturation curves for token budgets, scaling curves for agent count—and has begun to quantify them rigorously.

On T1, the evidence shows a genuine split. Hand-engineered structured memory (H-MEM, spatial DAGs, agent-specific updates) delivers measurable efficiency gains: role-aware context routing saves 25–47% of tokens without quality loss (Liu et al., 2025). But these approaches do not transfer across task domains (Yuen et al., 2025; Yu et al., 2026), and their efficiency saturates beyond ~2048 tokens per agent (Liu et al., 2025), suggesting they may hit an inherent ceiling imposed by heuristic design. Learned approaches like AgeMem's RL policy promise adaptability and potential bypass of saturation, but evidence for transfer or superiority is absent. Formal bounds on memory efficiency (Parakhin, 2026) suggest the future lies in combining learning with constraints—yet that combination has not been built or tested systematically.

On T2, evidence is equally unresolved. Multi-agent performance saturates due to coordination overhead, not long-context failure alone (Kim et al., 2025; Li et al., 2026; Yang et al., 2025; Mao et al., 2026), but the field does not know whether centralized coordinators or decentralized negotiation genuinely avoids this ceiling. The evidence is directional, not comparative. The interaction between memory capability and team size drives observed scaling limits, but the mechanism remains opaque beyond seven agents (Wu et al., 2026).

Critically, T1 and T2 are not independent. The choice of memory architecture determines what information is available for coordination and affects the feasibility of different coordination patterns. Conversely, the coordination protocol influences what memory must store and how it must be organized. A truly integrated design would co-optimize memory and coordination jointly, yet the literature has largely treated them as separate problems.

Two emerging failure modes—Semantic Intent Divergence and architectural drift—are hypothesized as root causes of enterprise system failures. Promising evidence comes from single frameworks and papers (Acharya, 2026; Rath, 2026), but causal mechanisms are speculative and untested via controlled ablation.

The verdict is that the field is transitioning from ad-hoc design toward principled trade-off navigation. It has developed the empirical apparatus to measure trade-offs and has begun to quantify them. But it has not yet isolated causal mechanisms, produced general solutions that transfer, or demonstrated how to co-optimize memory and coordination architectures. The path forward requires: (1) direct testing of learned memory policies on out-of-domain transfer, (2) controlled scaling experiments beyond seven agents to isolate whether coordination overhead or memory constraint drives the ceiling, (3) ablation studies separating the causal drivers of drift and semantic divergence, (4) task taxonomies predicting when centralization or decentralization is optimal, and (5) integration of learned policies with formal efficiency bounds. This last requirement is not exploratory: hybrid systems that combine adaptive learned routing with provable efficiency guarantees may overcome saturation ceilings that either approach alone cannot breach.

The problem lattice is not a weakness of the field—it is its current shape. The research ahead is not speculative; it is aimed at specific, measurable uncertainties about design choices that will determine whether multi-agent LLM systems can scale, transfer, and persist reliably in practice. Until those uncertainties are resolved, the trade-offs remain open, and design choices remain contested rather than established.
